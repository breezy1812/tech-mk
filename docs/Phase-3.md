# Phase 3 規劃書

## 專案名稱

**Local LLM Technical Knowledge Manager – Phase 3**

## 文件目的

在 Phase 2 已完成「文件可索引、問題可查詢、答案可回傳」的基礎上，Phase 3 的目標不是再重做主鏈路，而是針對**品質、可維運性、權限控制、資料同步與產品可用性**做持續優化。

這份文件聚焦在：

* 接下來有哪些值得做的優化項目
* 每個項目想解決什麼問題
* 實作上可以怎麼落地
* 完成後預期會帶來什麼效果

---

## 0. 目前基線

截至 Phase 2 結束，系統已具備：

* 本地文件載入與切塊
* Ollama embedding 與 chat model 串接
* Chroma 向量索引
* `/rag/reindex`、`/rag/status`、`/rag/query`
* Telegram `/askdoc`、`/ragstatus`、`/reindex`
* answer + sources + debug retrieved chunks
* 基本自動化測試與手動驗證文件

目前已在 Phase 3 先完成的項目：

* 增量索引與檔案同步

因此 Phase 3 的定位應該是：

> **從「可用的單路徑 RAG」走向「更準、更穩、更可控、更適合長期運行的知識系統」。**

---

## 1. Phase 3 核心目標

Phase 3 建議聚焦四件事：

1. 提升查詢品質
2. 降低人工維運成本
3. 強化權限與可觀測性
4. 提升實際使用體驗

換句話說，Phase 2 解決的是「有沒有」，Phase 3 要解決的是：

* 找得準不準
* 回得穩不穩
* 系統好不好維護
* 使用者敢不敢依賴它

---

## 2. 建議優先順序

### P3-A：先做的項目

這一批最值得先做，因為會直接改善查詢品質與日常使用穩定性：

* 檢索品質優化
* 增量索引與檔案同步
* 查詢觀測與評估機制
* 權限與文件範圍控制

### P3-B：第二批項目

這一批會強化產品體驗，但不一定要先做：

* 回答品質控制與結構化輸出
* 更完整的文件支援與解析策略
* Telegram / API 使用體驗優化

### P3-C：延伸性項目

這一批比較偏平台化與中長期能力：

* Web 後台或管理介面
* 工作流 orchestration
* 多知識庫 / 多租戶
* 多 agent 或工具呼叫

## 3. 優化項目清單

## 3.1 檢索品質優化

### 功能

讓 `/rag/query` 找回來的 chunks 更相關，降低答非所問、抓到雜訊文件或相關性不足的情況。

### 實作

建議分三層逐步做：

1. `top_k` 與 chunking 參數校調
2. 增加 query rewrite 或 query normalization
3. 導入 reranker 或 hybrid retrieval

可以先從低成本調整開始：

* 為不同問題類型測試 `top_k=3/5/7`
* 對 chunk size / overlap 做小規模 benchmark
* 加入 query rewrite，例如把口語問題轉成較穩定的檢索查詢

進一步再做：

* keyword + vector 的 hybrid retrieval
* cross-encoder reranker
* 依 score threshold 決定是否拒答

### 預期效果

* 提高命中正確文件的機率
* 降低檢索到不相干 chunk 的比例
* 讓回答品質更穩定
* 減少模型根據弱上下文硬湊答案的情況

### 優先級

**高**

---

## 3.2 增量索引與檔案同步（已完成）

### 功能

目前每次 reindex 都是全量重建。Phase 3 可改進成只更新新增、修改、刪除的文件，降低重建成本。

### 目前完成內容

目前已完成以下能力：

* 新增 indexing manifest，保存每份文件的 `file_hash`、`last_modified_at`、`chunk_ids`、`chunk_count`
* 新增增量同步邏輯，只對新增或已變更文件重新做切塊、embedding 與 upsert
* 新增刪檔同步，當文件已從 docs root 移除時，會同步清除對應 chunks
* 保留原本的全量 `/rag/reindex`，並新增 `/rag/sync` 作為增量同步入口
* sync report 可區分 `indexed`、`updated`、`unchanged`、`deleted`、`failed`
* 已補齊自動化測試，驗證更新檔、未變更檔與刪除檔情境

### 實作

目前實作採用檔案狀態管理：

* 為每份文件保存 `file_hash`、`last_modified_at`、`chunk_ids`、`chunk_count`
* sync 時先比對是否變更
* 僅對變更檔案重新切塊與 upsert
* 對已刪除檔案從 collection 中清除對應 chunks

若要更進一步，可以加：

* background indexing job
* 定時掃描 docs root
* 手動全量重建與增量同步分離成兩個 endpoint

### 預期效果

* 大幅減少 reindex 時間
* 降低 embedding 與 I/O 成本
* 讓知識庫更適合持續更新的真實使用情境
* 讓 docs root 的新增、修改、刪除能和向量索引維持一致

### 優先級

**已完成**

## 3.3 回答品質控制

### 功能

讓系統不只回答案，還能更一致地表達不確定性、引用依據與回答格式。

### 實作

可加入以下控制：

* 最低 relevance threshold，低於門檻時直接拒答
* 強制回答模板，例如：結論、依據、限制
* 回答附來源片段摘要，而不是只列檔名
* 支援 structured output，例如 JSON mode
* 針對不同問題型態切換回答風格

也可以把 RAG answer 分成兩層：

* 對使用者顯示的精簡答案
* 對 debug / 管理端顯示的完整 retrieval 細節

### 預期效果

* 回答更一致、更易讀
* 使用者更容易理解答案是根據什麼得出的
* 降低「看起來很像對，但其實依據不足」的風險

### 優先級

**中高**

---

## 3.4 評估與可觀測性

### 功能

讓系統不只「感覺可用」，而是能量化地知道哪些查詢做得好、哪些做得差。

### 實作

建議新增兩種能力：

1. 查詢觀測
2. 品質評估

查詢觀測可先做：

* 記錄每次 query 的 latency
* 記錄 top-k score 分布
* 記錄是否命中 sources
* 記錄是否走到拒答路徑

品質評估可先做：

* 建一份固定的測試題集
* 為每題標記預期文件與關鍵答案點
* 定期跑 retrieval hit rate 與 answer quality 回歸測試

### 預期效果

* 調參時有客觀依據
* 出問題時更快定位是 retrieval 還是 generation
* 可以逐步建立真實可追蹤的品質基線

### 優先級

**高**

---

## 3.5 權限與文件範圍控制

### 功能

避免所有人都能查所有文件，讓不同使用者或不同入口只看到被允許的資料範圍。

### 實作

初步可先做輕量版本：

* 文件 metadata 加上 `scope` 或 `group`
* query 時帶入 scope filter
* Telegram user id 對應可查詢的 scope
* `/reindex` 與查詢權限分開管理

進一步可演進為：

* 多知識庫 collection
* 部門 / 專案別隔離
* audit log

### 預期效果

* 避免資料誤洩漏
* 更接近企業內部系統的真實需求
* 後續做多租戶或部門知識庫會更順

### 優先級

**高**

---

## 3.6 文件支援與解析能力提升

### 功能

增加更多實務上常見文件格式的可用性，並降低 PDF / docx 解析品質不穩的情況。

### 實作

建議分兩段：

1. 先補主流辦公文件
2. 再補難處理文件

可先擴充：

* `.xlsx`
* `.csv`
* `.pptx`

再視需求加：

* 掃描 PDF OCR
* 表格抽取
* 文件結構保留，例如章節、標題、頁碼

### 預期效果

* 知識庫來源更完整
* 文件可用率提高
* 降低因 parser 品質不穩造成的 retrieval 偏差

### 優先級

**中**

---

## 3.7 使用體驗優化

### 功能

讓 API 與 Telegram 使用起來更自然，不只是能用而已。

### 實作

可優化的方向包括：

* `/askdoc` 支援更明確的錯誤提示
* `/ragstatus` 顯示更友善的摘要
* `/rag/query` 支援 `mode`，例如 `brief`、`detailed`
* 回答中顯示更易理解的來源格式
* 支援 streaming response

若之後要補 UI，可增加：

* Web 查詢頁面
* 文件狀態頁
* reindex 管理頁

### 預期效果

* 降低使用門檻
* 提升非技術使用者可理解性
* 增加日常使用黏著度

### 優先級

**中**

---

## 3.8 知識更新與營運工具

### 功能

把目前偏工程師手動操作的流程，逐步變成可維運的營運工具。

### 實作

可新增：

* reindex job 狀態追蹤
* indexing history
* failed file retry
* 文件清單與錯誤報表查詢 API
* 定時健康檢查與告警

### 預期效果

* 遇到 parser 或 embedding 問題時更容易排查
* 不必完全依賴 terminal 手動檢查
* 讓系統能更穩定地長期運行

### 優先級

**中高**

## 4. 建議實作順序

若要把 Phase 3 拆成幾個可交付的小階段，建議如下：

### Phase 3A：查詢品質與可觀測性

優先做：

* 檢索品質優化
* 拒答 threshold
* query log / latency / score metrics
* 固定測試題集與評估流程

完成後預期：

* 回答品質更穩
* 問題比較容易量化與回歸測試

### Phase 3B：資料同步與權限控制

優先做：

* 增量索引
* 刪檔同步
* scope filter
* user-to-scope 權限規則

完成後預期：

* 知識庫更適合真實更新情境
* 權限模型開始可用

### Phase 3C：產品化與營運工具

優先做：

* 營運報表
* failed file retry
* 更完整的文件支援
* Web 管理介面或查詢頁

完成後預期：

* 系統從工程原型更接近可持續使用產品

---

## 5. 不建議一開始就做的事情

雖然下面這些方向未來可能有價值，但不建議在 Phase 3 一開始就同時展開：

* 多 agent
* 複雜 workflow engine
* 過早導入太多模型切換策略
* 大量前端 UI 開發
* 一開始就做完整多租戶平台

原因很簡單：

目前最有價值的工作，仍然是先把**retrieval 品質、資料更新流程、權限邊界與評估能力**做好。這些是後續所有延伸功能的基礎。

---

## 6. 建議結論

如果只選最值得先做的四件事，建議是：

1. 檢索品質優化
2. 增量索引
3. 評估與可觀測性
4. 權限與 scope 控制

這四項完成後，系統會從目前的「可用 RAG 原型」提升成「可以更放心持續投入資料、持續調整、持續服務真實使用者」的版本。

Phase 3 的本質不是再多做幾個 endpoint，而是讓這套知識系統變得：

* 更準
* 更穩
* 更可維護
* 更能支撐真實使用情境
